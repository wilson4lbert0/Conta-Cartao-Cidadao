# SISTEMA DE PAGAMENTOS DIGITAIS — id.gov.cv (Cabo Verde)

## CONTEXTO DO PROJETO

Estou a desenvolver um módulo de pagamentos digitais integrado 
na aplicação oficial do Cartão Cidadão Digital de Cabo Verde 
(id.gov.cv), desenvolvida pelo NOSI (Núcleo Operacional para a 
Sociedade de Informação), disponível em:
https://play.google.com/store/apps/details?id=cv.gov.nosi.id

A aplicação já existe e está em produção. O objetivo é adicionar 
um módulo de pagamentos nativamente dentro desta app, sem criar 
uma aplicação separada. O backend de core banking será o 
Apache Fineract (open source, licença Apache 2.0).

---

## ARQUITETURA FUNDAMENTAL

### Identificação do Cliente
- O NÚMERO DE TELEMÓVEL do cidadão é o identificador único 
  da conta/cartão dentro do sistema
- Cada número de telemóvel = 1 wallet/conta digital
- O NIF (Número de Identificação Fiscal) serve como 
  identificador secundário e de verificação KYC
- A identidade já está verificada pelo sistema id.gov 
  (biometria + documentos do Estado)

### Estrutura da Conta Mãe (Float Account)
- Existe 1 CONTA MÃE central, titulada pelo NOSI, 
  alojada num banco parceiro cabo-verdiano (ex: BCA ou BCN)
- Cada carregamento feito por qualquer cidadão entra 
  nesta conta mãe
- O Apache Fineract gere os saldos individuais de cada 
  número de telemóvel como sub-contas virtuais
- INVARIANTE ABSOLUTO: 
  Soma de todos os saldos virtuais = Saldo da Conta Mãe
- O banco parceiro nunca interage diretamente com 
  o cidadão — apenas o NOSI gere os saldos virtuais

## FUNCIONALIDADES A IMPLEMENTAR

### 1. MÓDULO WALLET (Ecrã Principal de Pagamentos)
Criar um novo separador/tab "Pagamentos" dentro da app 
id.gov existente com:

- Visualização de saldo atual (em ECV — Escudo Cabo-Verdiano)
- Últimas 5 transações resumidas
- 4 botões de ação principais:
  [Pagar] [Receber] [Carregar] [Histórico]
- Número de telemóvel visível como identificador da conta
- Avatar/nome do cidadão puxado do perfil id.gov existente

### 2. MÉTODOS DE CARREGAMENTO (Cash-In)

Implementar os seguintes canais:

a) REDE DE AGENTES
   - App separada para agentes autorizados
   - Agente insere nº telemóvel do cliente
   - Agente confirma valor em dinheiro físico recebido
   - Sistema debita do float do agente e credita na wallet do cliente
   - Notificação push imediata para o cliente

b) VOUCHER / CÓDIGO DE RECARGA
   - Sistema de geração de códigos de 16 dígitos
   - Vendidos em lojas parceiras (como recargas de telemóvel)
   - Cliente insere código na app → saldo carregado
   - Funciona offline até sincronização

c) ATM VINTI4 (rede interbancária cabo-verdiana)
   - Integração via API com a SISP (Sociedade Interbancária 
     e Sistemas de Pagamentos)
   - Opção "Carregar Wallet id.gov" nos ATMs existentes
   - Input: número de telemóvel + valor + confirmação PIN

d) TRANSFERÊNCIA BANCÁRIA / OPEN BANKING
   - Para cidadãos com conta bancária
   - IBAN virtual associado ao número de telemóvel
   - Transferência bancária normal → carrega wallet automaticamente

e) REMESSAS INTERNACIONAIS (DIÁSPORA)
   - API webhook para receber de plataformas parceiras
   - (Western Union, Remitly, WorldRemit, etc.)
   - Destinatário identificado pelo número de telemóvel CV

### 3. MÉTODOS DE PAGAMENTO (Cash-Out / Pay)

a) QR CODE DINÂMICO
   - Comerciante gera QR com valor na app de agente/comerciante
   - Cliente abre id.gov → Pagar → Aponta câmara
   - Valor pré-preenchido automaticamente
   - Confirmação por biometria (impressão digital — já 
     integrada no id.gov)
   - Transação confirmada em < 3 segundos

b) QR CODE ESTÁTICO (para pequenos comerciantes)
   - QR fixo vinculado ao número de telemóvel do comerciante
   - Cliente lê QR e insere valor manualmente
   - Comerciante recebe notificação SMS + push

c) NFC (Near Field Communication)
   - Pagamento por aproximação entre dois telemóveis
   - Ou aproximação a terminal NFC parceiro
   - Limite: 5.000 ECV sem PIN adicional

d) USSD (*321#) — Para telemóveis básicos sem internet
   - Menu USSD em português e crioulo cabo-verdiano
   - Fluxo: *321# → 1.Pagar → NIF/Tel → Valor → PIN
   - Funciona em redes 2G (crítico para ilhas remotas)
   - Confirmação por SMS

e) REFERÊNCIA NUMÉRICA (para serviços e faturas)
   - Compatível com sistema de referências existente em CV
   - Entidade + Referência + Valor (ou valor automático)
   - Serviços: ELECTRA, EAGECV, impostos, propinas, etc.

f) TRANSFERÊNCIA P2P (entre pessoas)
   - Por número de telemóvel
   - Por NIF
   - Por QR Code pessoal
   - Por pedido de pagamento (request money)

g) PAGAMENTO ONLINE (e-commerce)
   - SDK/API para integração em lojas online
   - Checkout: "Pagar com id.gov"
   - Redirect para app → confirmação biométrica → retorna

### 4. SISTEMA DE AUTENTICAÇÃO E SEGURANÇA

Regras de autenticação por valor:

| Valor da transação | Método de autenticação |
|---|---|
| Abaixo de 500 ECV | PIN de 4 dígitos |
| 500 ECV — 5.000 ECV | Biometria (impressão digital) |
| 5.000 ECV — 50.000 ECV | Biometria + SMS OTP |
| Acima de 50.000 ECV | Biometria + SMS OTP + confirmação manual |

- Sessão da app: timeout de 5 minutos de inatividade
- Máximo de 3 tentativas de PIN/biometria → bloqueio temporário
- Todas as transações registadas com timestamp, localização 
  (opcional, com consentimento) e device ID

### 5. NÍVEIS DE CONTA (KYC Progressivo)

NÍVEL 1 — Conta Básica (KYC mínimo)
- Requisito: número de telemóvel verificado + selfie
- Saldo máximo: 25.000 ECV
- Transação máxima: 5.000 ECV
- Limite mensal: 50.000 ECV

NÍVEL 2 — Conta Intermédia
- Requisito: NIF + BI/Passaporte verificado via id.gov
- Saldo máximo: 150.000 ECV
- Transação máxima: 50.000 ECV
- Limite mensal: 300.000 ECV

NÍVEL 3 — Conta Completa
- Requisito: KYC completo (já feito pelo id.gov) + 
  ligação a conta bancária
- Sem limites especiais além dos regulatórios do BCV
- Open banking ativo

### 6. GESTÃO DE AGENTES

- Dashboard web para gestão de agentes autorizados
- Cada agente tem float próprio (liquidez pré-carregada)
- Sistema de reposição de float via banco
- Comissão automática por transação (configurável)
- Relatórios de performance por agente
- Alertas de float baixo

### 7. NOTIFICAÇÕES

- Push notification imediata para cada transação
- SMS como fallback quando sem internet
- Formato: "Pagou X ECV a [Nome/Loja]. Saldo: Y ECV"
- Recibo digital gerado automaticamente (PDF/partilhável)

## STACK TECNOLÓGICO

### Backend
- Apache Fineract 1.14.0 (core banking engine)
  https://fineract.apache.org
- API REST para comunicação com a app
- Autenticação: OAuth 2.0 + JWT
- Base de dados: PostgreSQL
- Cache: Redis (para saldos e sessões)
- Filas: RabbitMQ ou Apache Kafka (para transações)

### Frontend (módulo a adicionar no id.gov)
- Framework atual do id.gov (adaptar ao existente)
- Se Flutter: adicionar módulo payments como feature module
- Se React Native: adicionar stack de navegação separado
- Se nativo Android/iOS: adicionar activity/viewcontroller

### Segurança
- TLS 1.3 para todas as comunicações
- Encriptação AES-256 para dados em repouso
- Certificado pinning na app mobile
- Assinatura de transações com chave privada do dispositivo
- Logs de auditoria imutáveis para conformidade BCV

### Integrações Externas
- SISP/Vinti4: API de integração ATM
- CVTelecom/T+: USSD gateway e SMS
- STII (Sistema de Transferências Imediatas — BCV): 
  quando disponível via API
- Apache Fineract: todas as operações de core banking

## CONFORMIDADE REGULATÓRIA (BCV)

O sistema deve gerar automaticamente:

- Relatório diário de transações para o Banco de Cabo Verde
- Relatório de reconciliação Conta Mãe vs saldos virtuais
- Alertas de transações suspeitas (AML/KYC)
- Relatório de agentes e volumes por ilha/município
- Exportação em formato compatível com RJSPME 
  (Decreto-lei 8/2018)

Limites regulatórios a implementar conforme RJSPME:
- Monitorização de transações acima de 100.000 ECV
- Reporte obrigatório ao BCV de transações suspeitas
- Arquivo de logs por mínimo 5 anos

## FLUXO COMPLETO DE EXEMPLO

### Cidadão paga 1.500 ECV no mercado via QR:

1. Comerciante abre app → gera QR com valor 1.500 ECV
2. Cliente abre id.gov → tab Pagamentos → Pagar → Câmara
3. App lê QR → mostra "Pagar 1.500 ECV a Mercado Silva?"
4. Cliente confirma com impressão digital
5. id.gov → API NOSI → Apache Fineract
6. Fineract: debita 1.500 ECV da sub-conta do cliente
7. Fineract: credita 1.500 ECV na sub-conta do comerciante
8. [Conta Mãe não muda — é movimento interno]
9. Cliente recebe push: "Pagou 1.500$ a Mercado Silva. 
   Saldo: 8.500$"
10. Comerciante recebe push: "Recebeu 1.500$ de J. Tavares"
11. Transação gravada com UUID único para auditoria BCV

## NOTAS IMPORTANTES

- O número de telemóvel no formato cabo-verdiano: 
  +238 seguido de 7 dígitos (ex: +238 9XXXXXX)
- Moeda: ECV (Escudo Cabo-Verdiano), 
  código ISO 4217: CVE
- Idiomas: Português (principal) + Crioulo Cabo-Verdiano
- Fuso horário: UTC-1 (Cabo Verde)
- O id.gov já tem biometria implementada — 
  reutilizar sem reimplementar
- Privacy by design: dados mínimos, 
  sem partilha com terceiros (conforme política actual do app)
- O NOSI é entidade pública empresarial — 
  conformidade com lei cabo-verdiana de proteção de dados
